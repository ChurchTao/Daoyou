# 造物引擎 v2 词条库重构方案 (Daoyou V2 Affix System Refactoring Plan)

## 1. 核心设计哲学：平衡三角论 (The Balanced Triangle) & 维度切分模式 (Dimensional Split)

为了根除当前词缀定义中功能混杂（如反伤、吸血在三者中均有出现）的问题，并在《万界道友》修仙世界观下构建出深度联动的战斗体验，我们将对三大产物（技能、功法、法宝）进行严格的功能边界划分。

三者缺一不可，互相依存，构成“平衡三角”。

### 1.1 严格的物理边界隔离 (Strict Exclusive Isolation)

所有词缀必须严格遵循其所属产物的物理边界，不存在任何形式的跨界。

| 产物类型 | 定位与核心作用 | 允许的 GE 类型 (Effect Template) | 核心词缀形态 | 随机前/后缀形态 |
| :--- | :--- | :--- | :--- | :--- |
| **法宝 (Artifact)** | **磐石与底线** (容错与面板) | `attribute_modifier` (固定值), `shield`, `reflect`, `damage_immunity` | **固定面板属性** (All-in-Affix，提供固定血量/攻击/防御) | **受击/防守触发** (OWNER_AS_TARGET，如濒死保命、概率免伤、被动反伤) |
| **功法 (Gongfa)** | **杠杆与规则** (流派与倍率) | `attribute_modifier` (百分比), `percent_damage_modifier`, `cooldown_modify`, `resource_drain` (非瞬时) | **全局百分比属性** (提供百分比攻击/防御/血量倍率) | **全局机制修改** (OWNER_AS_CASTER 或全局事件，如减CD、全局增伤、击杀/行动回蓝) |
| **技能 (Skill)** | **锋刃与变数** (交互与爆发) | `damage`, `heal`, `dispel`, `apply_buff` (强力短效状态), `mana_burn` | **主动直接效果** (如造成X点伤害、恢复X点生命) | **瞬发增幅/附加状态** (如附加灼烧、触发破甲、本次技能爆伤翻倍) |

## 2. 代码结构的物理重组 (Physical Separation)

### 2.1 废除 `commonAffixes.ts`
当前的 `commonAffixes.ts` 虽然减少了代码量，但模糊了三大产物的边界。
**行动项**：
*   删除 `commonAffixes.ts` 文件。
*   将其中的通用属性按类别强行拆解并重新分发到对应的专属文件中。
    *   固定值加成（如固定体魄、固定物防）迁移至 `artifactAffixes.ts`。
    *   百分比加成（如灵力提升10%、全伤害加成）迁移至 `gongfaAffixes.ts`。
    *   **优点**：代码完全隔离，后续调整“法宝的体魄”和“功法的体魄”互不干涉，从代码层面杜绝越界。

### 2.2 产物生成的模板约束 (Template Constraints)
在 `Creation Process`（造物流程）中，增加基于 `ProductType` 的词缀分类过滤机制。
*   在抽取词缀时，引擎将基于产物类型读取其专属文件中的定义，从根本上阻止法宝抽到功法的词缀。
*   这保障了“词缀全包揽 (All-in-Affix)”原则，即法宝的基础固定属性和受击特效都通过它专属的词缀池来实现。

## 3. 构建多维共鸣机制 (Resonance Mechanisms)

在严格的功能隔离下，为了实现“1+1+1 > 3”的三角平衡联动，我们将在词缀系统（特别是 `conditions` 和 `listenerSpec` 中）深度集成以下共鸣能力。

### 3.1 标签检查共鸣 (Tag Check Synergy)
通过 `GameplayTags`（如元素标签、特性标签）让三个隔离的系统互相通信。
*   **示例 (火系流派)**：
    *   **技能**：带有 `[火]` 标签，造成基础火系伤害。
    *   **功法**：`if (ability_has_tag('火')) { 增加 30% 火系伤害倍率 }`。
    *   **法宝**：`if (owner 施放的技能 has_tag('火')) { 为自身附加相当于该次伤害 10% 的护盾 }`。

### 3.2 状态链条共鸣 (Status Chain Synergy)
通过战斗中的临时状态（Buff/Debuff）作为桥梁，实现战术级别的见招拆招。
*   **示例 (冰霜碎冰流)**：
    *   **技能**：命中时 80% 概率为目标挂上 `[冰缓]` (Status) 状态。
    *   **功法**：`if (target.has_status('冰缓')) { 攻击时额外附带目标最大生命值 2% 的真实伤害 }`。
    *   **法宝**：`if (target.has_status('冰缓')) { 自身受击时 50% 概率触发绝对防御 }`。

### 3.3 数值门槛隐性共鸣 (Numerical Threshold, 预留能力)
虽然我们主要依赖标签和状态，但基于条件检查器（Conditions），我们天然支持数值突破带来的质变。
*   **示例**：功法中配置条件 `if (owner.DEF > 2000)`，触发额外的 15% 减伤。（这里的 2000 DEF 必然需要极品法宝的固定数值来支撑，从而实现隐性联动）。

## 4. 重构实施路径

1.  **清退与拆解**：移除 `commonAffixes.ts`，将其内容按规则（固定值给法宝，百分比给功法）瓜分。
2.  **法宝词缀净化 (`artifactAffixes.ts`)**：清理所有增加百分比属性的词缀；清理所有主动攻击的特效（如造成伤害吸血）；强化 `shield`、`reflect` 和 `damage_immunity` 相关的受击词缀。
3.  **功法词缀净化 (`gongfaAffixes.ts`)**：清理所有固定数值属性（如+10点攻击）；清理受击触发的防御特效；将所有的全局机制（如回合回血、CD缩减、全局伤害增幅）整合在此。
4.  **技能词缀净化 (`skillAffixes.ts`)**：清理所有常驻属性提升（如进入战斗加 10% 攻击）；确保所有前缀后缀都只针对“本次技能施放”生效（如挂状态、本次伤害吸血、本次技能减CD）。
5.  **共鸣条件植入**：在各个重构后的词缀文件中，大量运用 `ability_has_tag` 和 `has_tag` 条件，构建“标签流派”和“状态流派”的预设词缀。

## 5. 预期结果
重构完成后，玩家在造物和搭配时将面临清晰的抉择：**必须通过炼制法宝堆砌基础面板和容错率，必须领悟功法来确立流派和倍率，必须创造技能来打出核心连招。三者各司其职，又在标签和状态的串联下形成极致的战斗共鸣。**
