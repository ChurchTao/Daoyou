# 无相禅宗分层形态重构设计稿

## 1. 文档状态

- 状态：已实施，待玩法验收
- 日期：2026-07-21
- 适用范围：无相禅宗定义、明镜照业、魔心渡厄、36 个参悟节点、battle-v5 动态技能形态、技能展示、AI 选招、测试与代码清理
- 当前配置版本：`2`
- 目标配置版本：`2`
- 核心决策：佛相、魔相、无相继续作为每门神通真实可见的技能形态；实现统一收敛为 `A / A+B / A+B+C` 的效果分层
- 数值状态：本文数值是首轮实施基线
- 文档权威：实施期间，如旧代码、旧测试、旧展示文案与本文冲突，以本文为重构目标；数据库稳定 ID 和通用 battle-v5 既有消费者除外

## 2. 执行摘要

无相禅宗的最大特色不是单一战意资源，也不是一个持续两回合的增益，而是同一门神通在佛相、魔相、无相中呈现逐层展开的不同面貌。重构不得把魔相和无相降级为两层全局 Buff。

本设计保留以下核心体验：

- 佛相是每门神通的完整主体效果 `A`。
- 魔相在主体上追加该技能专属效果 `B`，最终为 `A+B`。
- 无相继承魔相，再追加该技能专属效果 `C`，最终为 `A+B+C`。
- 3～5 点战意开启两门魔相神通。
- 6 点战意开启一门无相神通。
- 每门技能继续拥有独立的佛相名、魔相名、无相名和实时描述。
- 明镜照业继续围绕业痕、承受、反击与现报。
- 魔心渡厄继续围绕主动沉血、低血强化、短窗减伤、控制免疫与吸血。

本设计删除以下实现方向：

- 魔相第一式、第二式分别拥有完全不同的技能树。
- 同一技能在不同形态改变目标阵营或目标策略。
- 递归扫描任意 `EffectConfig` 并自动推断“强化 15%”如何作用。
- 任意 Buff 在单位之间复制或转移。
- 延迟伤害、真实劫债、虚拟劫债和不消费债务返射同时存在。
- 记录第一门魔相神通 ID，再比较第二门是否不同。
- 为六门神通复制“下一次费用减半/增加”的高优先级变体。
- 无相自动合并所有入魔条款和渡厄条款。
- 为无相专门扩展 action budget、伤害分流或复杂伤害来源追踪。

battle-v5 获得的通用演进只有“技能效果层与效果计划”。它不知道佛、魔、无相，也不知道业痕、魔契、劫债。无相内容只通过通用配置组合基础效果和追加效果。

## 3. 当前实现为何需要重构

### 3.1 完整变体数量膨胀

当前实现为两条道途的六门神通分别编译佛相、魔相第一式、魔相第二式和无相式，形成 48 格实际结算矩阵：

```text
2 条道途 × 6 门神通 × 4 套形态 = 48 套完整行为
```

每套行为可以独立覆盖：

- 名称
- 描述
- 气血成本
- 目标策略
- AI 意图
- 主效果
- 施法效果
- 形态推进

这使技能形态不再是“同一技能的变化”，而是借用同一个技能 ID 承载四门不同技能。

### 3.2 单宗门反向扩建 battle-v5

当前无相直接或间接推动了以下通用层能力：

- `DynamicDataDrivenActiveSkill`
- `AbilityVariantConfig`
- `ability_mode`
- `ability_mode_ability_differs`
- `ability_variant_is`
- `status_transfer`
- `damage_cap`
- `lifesteal.variantIds`
- `damage_memory.event = ability_cost_paid`
- `damage_memory.consumeRatio`
- `damage_memory.maxReleaseValue`
- `scaleNumericEffectsByLayer`
- `effectStrengthScaler`

其中多项没有其他生产内容消费者，或者只有无相要求当前这套参数组合。单宗门不应要求 battle-v5 永久维护对所有效果类型的递归强度解释。

### 3.3 节点开始修改元规则

当前部分节点不是给技能增加效果，而是在修改“效果如何解释”：

- 所有入魔条款提高 15%。
- 所有渡厄条款提高 20%。
- 无相将两组条款按 80% 或 100% 合并。
- 按实际消费层数递归强化嵌套效果。
- 第二门必须与第一门不同才改变费用。

这类节点会随着新 Effect 类型增加而持续扩张维护面，且很难从玩家文案直接推导真实结算。

### 3.4 内容结构违反新节点扩展边界

当前 `WUXIANG_FEATURES: Set<string>` 与集中式 `features.has(...)` 让全部节点回流到单个大型编译器。节点定义和节点行为分离，且新节点必须修改中央文件。

重构后，节点必须通过流派私有 Build Facade 修改语义字段，或直接作为节点插件装饰对应技能层。不得继续扩大中央 `switch + features.has`。

## 4. 目标与非目标

### 4.1 目标

- 保住佛相、魔相、无相作为门派第一识别特征。
- 保住每门神通独立的魔相和无相变化。
- 让形态变化具有明确的继承关系，而不是相互替换。
- 让目标、费用、冷却和主体定位在三相间稳定。
- 让 battle-v5 的新增能力对其他技能体系可复用。
- 删除只有无相使用且已不再需要的底层机制。
- 保持现有宗门、流派、心法、技能、节点和战术 ID 稳定。
- 允许逐阶段重构，同时保证每个阶段有明确测试门禁。

### 4.2 非目标

- 本次不重做宗门组织、任务、晋升、设施、经济和地图。
- 本次不修改五项角色基础属性边界。
- 本次不持久化 `battleProjection`、形态、业痕、战意或战斗 Buff。
- 本次不重做 creation-v2 的词缀体系。
- 本次不为所有现有技能迁移到效果层；首个生产消费者是无相。
- 本次不以保留旧战斗录像逐事件一致为目标。
- 本次不保留旧无相行为兼容开关。

以下 ID 必须保持不变：

- 宗门：`wuxiang`

## 6. 术语与不可变契约

### 6.1 三层效果

```text
A：佛相主体层
B：魔相追加层
C：无相追加层
```

解析结果：

```text
佛相 = A
魔相 = A + B
无相 = A + B + C
```

### 6.2 效果执行顺序

一门技能的执行顺序固定为：

```text
资源支付
-> 施法前效果 castEffects
-> A 主效果
-> B 主效果（若启用）
-> C 主效果（若启用）
-> A 完成效果
-> B 完成效果（若启用）
-> C 完成效果（若启用）
-> 成功消费形态次数
```

“完成效果”用于保证心隙、业门等状态在本次所有伤害段结束后再施加，避免同一技能的 B/C 提前消费刚由 A 施加的状态。

### 6.3 形态层硬约束

- B、C 只能追加，不能删除、替换或重新排序 A。
- 三相共享同一 `targetPolicy`。
- 三相共享同一技能气血成本。
- 三相共享同一冷却。
- 三相共享同一基础技能标签。
- B、C 不得提供新的目标策略。
- B、C 不得覆盖 AI `selectionProfile`。
- B、C 每层最多声明两个主要机制；多段伤害仍视为一个伤害机制。
- B、C 不得递归修改任意子 Effect 的数值。
- 节点只能修改明确字段或追加一条明确效果。
- `shouldApplyEffects === false` 时不进入主效果/完成效果链，也不消费形态次数。
- 一旦 `executeSkill` 已进入主效果链，本次施法即视为成功结算：即使驱散、净化、条件治疗等原子效果因没有合法对象而 no-op，完成效果仍执行且形态次数仍消费。不得为此增加“至少一个 Effect 实际改值”的回执协议。
- 技能开始后冻结效果计划；结算期间状态变化不重选计划。
- 形态层不得直接读写持久化模型。

## 7. battle-v5 通用效果层设计

### 7.1 配置模型

建议新增通用配置，而不是继续扩大 `AbilityVariantConfig`：

```ts
export interface AbilityEffectLayerConfig {
  id: string;
  effects?: EffectConfig[];
  completionEffects?: EffectConfig[];
}

export interface AbilityEffectPlanConfig {
  id: string;
  name: string;
  description?: string;
  priority: number;
  conditions: ConditionConfig[];
  layerIds: string[];
  consumeModeKey?: string;
}
```

`AbilityConfig` 增加：

```ts
effects?: EffectConfig[];
completionEffects?: EffectConfig[];
effectLayers?: AbilityEffectLayerConfig[];
effectPlans?: AbilityEffectPlanConfig[];
```

基础 `effects/completionEffects` 即 A。无相技能编译为：

```ts
effectLayers: [
  { id: 'demon', effects: B, completionEffects: BDone },
  { id: 'formless', effects: C, completionEffects: CDone },
],
effectPlans: [
  {
    id: 'formless',
    name: '无相式名称',
    priority: 300,
    conditions: [formIs('formless')],
    layerIds: ['demon', 'formless'],
    consumeModeKey: WUXIANG_FORM_MODE,
  },
  {
    id: 'demon',
    name: '魔相式名称',
    priority: 200,
    conditions: [formIs('demon')],
    layerIds: ['demon'],
    consumeModeKey: WUXIANG_FORM_MODE,
  },
];
```

没有命中任何计划时使用基础佛相 A。

### 7.2 计划不是完整变体

效果计划允许改变：

- 展示名
- 展示描述
- 启用的效果层
- 成功后消费的通用 mode key

效果计划禁止改变：

- 费用
- 冷却
- 目标
- 选招意图
- 基础标签
- 技能 ID
- 解锁条件

### 7.3 运行时解析

新增纯函数：

```ts
resolveAbilityEffectPlan(config, context): ResolvedAbilityEffectPlan
```

返回：

```ts
interface ResolvedAbilityEffectPlan {
  id?: string;
  name: string;
  description?: string;
  effects: ReadonlyArray<EffectConfig>;
  completionEffects: ReadonlyArray<EffectConfig>;
  consumeModeKey?: string;
}
```

解析时必须：

- 按 `priority` 从高到低选中一个计划。
- 校验所有 `layerIds` 存在且不重复。
- 按计划声明顺序拼接层。
- 拼接全部主效果后，再拼接全部完成效果。
- 冻结解析结果进入施法快照。
- 禁止在结算中再次读取实时形态重选计划。

### 7.4 技能类调整

将 `DynamicDataDrivenActiveSkill` 收敛为 `LayeredDataDrivenActiveSkill`，或在原类内彻底替换完整 variant 语义。首选新类名，避免旧接口继续被误用。

职责：

- 固定目标、费用、冷却和 AI 意图。
- 施法准备时解析并冻结效果计划。
- 执行主效果链。
- 执行完成效果链。
- 主效果链与完成效果链正常执行后消费一次 mode；个别原子效果 no-op 不阻止消费。
- 对外暴露当前 `runtimePlanId`、显示名和描述。

不再负责：

- 按变体选择不同目标。
- 按变体解析不同费用。
- 按变体覆盖施法条件。
- 按变体覆盖 AI 意图。

### 7.5 Factory 校验

`AbilityFactory` 必须验证：

- layer ID 唯一。
- plan ID 唯一。
- plan 引用层存在。
- plan 不重复引用同一层。
- `consumeModeKey` 非空。
- 基础和全部层合并后仍满足 Ability 标签契约。
- 敌方攻击技能仍具有伤害功能标签。
- 自身防御技能不会因层追加敌方目标效果而改变 targetPolicy。
- 完成效果不能包含需要在命中前执行的资源支付。

### 7.6 展示与日志

- 技能栏显示当前计划名称和当前稳定气血成本。
- 技能详情按 `佛相 / 魔相追加 / 无相追加` 分段展示。
- 战斗快照记录 `runtimePlanId`，不再记录任意 variant 全量覆盖。
- 日志中的 ability ID 保持稳定，显示名使用冻结计划名称。
- UI 不自行拼接 B/C；展示与战斗必须读取同一个编译结果。

## 8. 形态运行态

### 8.1 保留的通用字段

```ts
export interface AbilityModeRuntime {
  key: string;
  mode: string;
  remainingUses: number;
  displayName: string;
  cleanupBuffIds?: string[];
}
```

### 8.2 删除的字段

- `phase`
- `firstAbilityId`

无相不再区分魔一、魔二，也不比较前后技能 ID。

这里的删除范围必须完整但只限 ability mode：同步删除 `AbilityModeRuntime.phase/firstAbilityId`、`AbilityModeParams.phase`、`ability_mode_is.params.phase`、ability-mode 状态视图中的 phase 展示，以及 `advanceAbilityMode` 的 phase 自增和 ability ID 记录参数。不得误删行动队列、战斗状态机或日志事件中同名但语义无关的 `phase` 字段。

### 8.3 消费时机

- 《一念未生》设置 `mode=demon, remainingUses=2` 或 `mode=formless, remainingUses=1`。
- 只有命中对应 effect plan、`shouldApplyEffects !== false` 且主效果链正常返回时才减少 `remainingUses`。
- 驱散无目标、净化无目标、条件治疗未达阈值等合法 no-op 仍算一次形态神通；否则自目标技能可通过空放无限保留形态。
- 普通攻击、装备技能、非无相宗门技能不消费该 mode。
- mode 耗尽时清理公共形态 Buff。
- 形态推进由 `LayeredDataDrivenActiveSkill` 根据 `consumeModeKey` 完成，不在每个 B/C 中手写 `ability_mode advance`。

## 11. 明镜照业最终设计

### 11.1 道途描述

> 以承受制造因，以魔相兑现果。来力不急于拒绝，只把每一道因果留在镜中；佛相立因，魔相现报，无相令因果同时照见。

### 11.2 核心被动《明镜照业》

- 业痕上限3层，持续整场战斗，不可普通驱散。
- 佛相中，每次敌方行动首次对自身造成至少1点直接伤害时，获得1层业痕。
- 佛相受到敌方直接伤害时，返还该次实际伤害的 `5% + 每层业痕3%`。
- 反伤只响应 `DIRECT`，不会响应反伤、追击、延迟伤害或气血成本。
- 每次成功施展六门佛相基础宗门神通之一获得1点战意；《一念未生》不产生战意。
- 魔相、无相中不因受击产生业痕，也不触发佛相即时反伤。
- 取消同一敌方行动共享最大气血比例反伤上限。

### 11.3 公共魔相护体《止观》

- 进入魔相后获得20%直接伤害减免。
- 持续到两门魔相神通成功结算。
- 无相施法期间同样获得止观，结算后立即清除。
- 止观不可普通驱散。

## 18. AI 选招规则

### 18.1 通用要求

- AI 候选目标只由稳定 `targetPolicy` 决定。
- AI 不因形态改变技能目标。
- 形态计划只改变技能显示名和效果，不改变 `selectionProfile`。
- 战术策略可以读取战意、业痕、形态、当前气血和冷却。
- 策略不得读取 runtime plan ID 枚举技能行为。

### 18.2 明镜优先级

魔相/无相攻击优先：

```text
三叩业门（目标已有业门）
拈花叩心
照见五蕴（目标有可驱散增益）
```

魔相/无相防御优先：

```text
闭目观劫
一苇横江
血海听潮
```

守镜战术在业痕不足3层时优先防御技能；现报战术在至少1层业痕时优先可消费业痕的攻击技能；无相战术优先积满6战意。

### 18.3 魔心优先级

低血防御优先：

```text
一苇横江
闭目观劫
血海听潮
照见五蕴（自身存在可净化减益）
```

魔相攻击优先：

```text
三叩业门（低于45%气血）
拈花叩心
已有血潮时选择任意可用攻击神通
```

无相优先三叩业门与拈花叩心；低于30%气血时允许选择一苇横江。

## 19. battle-v5 机制保留、演进与回退

### 19.1 保留

| 机制 | 决策 | 原因 |
| --- | --- | --- |
| 当前气血比例成本 | 保留 | 无相核心代价，且已是通用 `AbilityCostConfig` |
| 原子气血支付与 `AbilityCostPaidEvent` | 保留 | 阈值节点需要可靠的支付前后快照 |
| 施法快照 | 保留并扩展 | 分层计划必须在施法开始冻结 |
| `ability_mode` 基础运行态 | 保留并简化 | 可复用于姿态、充能与有限次数强化 |
| `ability_mode_is` 条件 | 保留 | 计划选择需要 |
| `consume_status_trigger` 基础消费/重复 | 保留 | creation-v2 与红尘剑宗仍有生产消费者 |
| `damage_memory` 基础记录/释放 | 保留 | 红尘剑宗和 creation-v2 仍使用 |
| `damage_defer` | 保留 | creation-v2 仍使用；无相不再使用 |
| listener trigger budget | 保留 | 每轮首次、Buff 生命周期触发次数是通用能力 |
| `damage_taken_at_least` 等事件条件 | 若已存在则保留 | 属于简单通用事件判断，不编码无相语义 |

### 19.2 通用演进

| 机制 | 调整 |
| --- | --- |

## 22. 测试设计

### 22.1 battle-v5 效果层测试

- 基础无计划只执行 A。
- 魔相计划执行 A+B。
- 无相计划执行 A+B+C。
- completion effects 在全部主效果之后执行。
- plan priority 确保无相优先于魔相。
- 相同技能三计划共享 targetPolicy、费用和冷却。
- 配置引用未知层时注册失败。
- 重复 layer ID、plan ID、layer 引用时注册失败。
- plan 不允许提供 costs、targetPolicy 或 selectionProfile。
- `shouldApplyEffects === false` 时不执行 completion effects、不消费 mode。
- 主效果中的驱散/净化/条件治疗 no-op 时仍执行 completion effects 并消费 mode。
- 施法准备后改变 mode 不改变当前快照。

### 22.2 无相基础矩阵

新矩阵为：

```text
2 道途 × 6 技能 × 3 形态 = 36 格
```

每格验证：

- 稳定技能 ID
- 当前显示名
- 固定目标
- 固定气血成本
- 固定冷却
- A 是否始终存在
- B 是否只在魔相/无相存在
- C 是否只在无相存在
- completion effect 顺序
- 战意变化
- mode 剩余次数

- 36节点逐项编译出唯一、可观察差异。

- 无节点调用中央 `features.has(nodeId)`。
- 终极节点不得改变 effect layer 组合规则。

### 22.4 关键行为测试

明镜：

- 只有敌方直接伤害产生业痕。
- 同一敌方行动只基础获得1层业痕。
- 魔相/无相不产生佛相受击业痕。
- B 无业痕时只不执行消费后效果，A仍完整结算。
- 免费现报不触发照还来处治疗。
- 业门在全部A伤害结束后施加。
- 倒叩在 A 完成效果之前原子消费旧业门，不引爆本次技能新生成的业门。
- 无相三叩的 C 只追加固定伤害，最终新业门层数仍由 A 与节点共同决定。
- 无相观劫可触发两次。

魔心：

- 气血成本不发布伤害事件。
- 三叩低血判断读取支付后快照。
- 魔相两门技能都执行相同B，不区分phase。
- 无相只执行一次A+B+C。
- 吸血按宗门机制标签，不按计划ID。
- 防御型魔相技能不错误触发直接伤害吸血。

## 25. 风险与处置

### 风险一：A+B+C 数值叠加过强

处置：只调整B/C显式系数、转相成本、战意阈值或公共护体参数，不改变层继承关系。

### 风险二：完成效果顺序引入通用复杂度

处置：完成效果只有单一语义——`executeSkill` 已进入且主效果链正常返回后，按层顺序执行。它不收集单个 Effect 的“是否改值”结果，不得扩展为任意阶段图或可编程流水线。

### 风险三：自目标技能的魔相缺少攻击感

处置：通过护盾、受击反击、下一击强化、净化和低血恢复体现专属B/C；不得临时把目标改为敌方。

### 风险四：节点又开始修改元规则

处置：节点审查必须回答“它修改了哪个明确字段或追加了哪一条明确效果”。无法回答的节点必须重写。

### 风险五：删除 Effect 影响 creation-v2

处置：删除前使用全仓生产代码搜索确认消费者。`damage_memory`、`damage_defer`、`consume_status_trigger.scaleEffectsByLayer` 已知有其他消费者，不得整体删除。

### 风险六：为未上线内容误加兼容复杂度

处置：只保留 v2 单一真相。开发环境旧数据直接重建；代码评审拒绝无相 v1 转换器、双版本判断、兼容开关、迁移 SQL和维护窗口逻辑。

## 27. 最终结论

无相禅宗的实现目标不是减少到“一个资源加两个 Buff”，也不是维持48套相互独立的技能。正确边界是：

```text
同一技能，稳定主体；
魔相添一层，无相再添一层；
层层可见，层层可解释，层层可测试。
```

本文将门派特色放在内容层的A/B/C设计中，将通用演进限制在 battle-v5 的效果层、计划解析、快照与完成效果四个边界内。这样既保住佛相、魔相、无相的核心幻想，也避免一个宗门继续迫使整个战斗架构理解业痕、魔契、劫债和彼岸等内容语义。
